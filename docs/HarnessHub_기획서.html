01 — Why Now · 왜 지금인가

하네스 엔지니어링은
학문이 되고 있다.

2024년부터 SWE-agent, OpenHands, Aider, Cline, METR ResearchAgent 같은 에이전트 하네스가 폭발적으로 등장했다. 모델보다 하네스가 성능을 좌우하는 시대다. 하지만 정작 이 하네스들은 GitHub 어딘가에 흩어져 있다.

VALUE 01
⌕
Discover
GitHub 전역에서 자동 수집한 하네스를 카테고리·프레임워크·모델별로 큐레이션한다.
VALUE 02
↓
Install
한 줄 명령어로 설치 가능. CLI · Docker · npm · pip 통합 제공.
VALUE 03
★
Evaluate
SWE-bench 등 표준 벤치마크 점수와 커뮤니티 리뷰를 한눈에 비교.
VALUE 04
⤴
Share
자신의 하네스를 등록하면 GitHub 푸시할 때마다 자동 동기화된다.
02 — Problem & Opportunity

하네스는 흩어져 있다,
가치는 잠겨 있다.

에이전트 하네스 생태계는 폭발적으로 자라지만, 발견 가능성·재현성·표준이 부재하다. "이런 게 있는지 몰랐어요"가 가장 많이 듣는 말이다.

12,400+
2025년 1년간 GitHub에 등장한 "agent" 토픽 신규 리포
SOURCE · GitHub Search API, 2025 추정
73%
"하네스 선택에 1주일 이상 걸렸다"고 답한 ML 엔지니어
SOURCE · 가상 설문, n=120 (실제 출시 시 검증 필요)
∅
현재 "AI 에이전트 하네스" 전용 큐레이션 마켓플레이스의 수
SOURCE · 경쟁 분석 (Claude Skills · MCP 마켓은 있지만 하네스는 없음)
⊕
발견 비용이 너무 높다
GitHub Trending, Awesome 리스트, 트위터, 블로그를 뒤져야 한다. 표준 디렉터리가 없다.
⊗
메타데이터가 제각각
어떤 모델 호환되는지, 어떤 도구 사용하는지, 라이선스가 무엇인지 — README를 다 읽어야 안다.
⊘
설치·실행 진입장벽
의존성, API 키, Docker 설정 등 환경 구성이 복잡. 한 줄로 시도해볼 방법이 없다.
⊜
평가·비교 어려움
"이 하네스가 정말 좋은가?"를 판단할 표준 메트릭이 없다. 별점만으론 부족하다.
03 — Target Personas

세 명의 개발자.

HarnessHub은 일반 개발자도 진입 가능하도록 설계된다. AI 박사학위는 필요 없다. 호기심과 한 줄 명령어 실행 능력만 있으면 된다.

04 — Solution Overview

GitHub과 동기화되는
큐레이션 마켓플레이스.

HarnessHub은 npm registry처럼 자동화되고, Vercel Templates처럼 큐레이션되며, Product Hunt처럼 커뮤니티가 살아있는 — 세 가지를 결합한 새로운 카테고리다.

README

SWE-agent는 LM(예: Claude 4 Sonnet)을 GitHub 이슈를 푸는 자율 소프트웨어 엔지니어로 변환합니다. 실제 코드베이스에서 작동하며, 코드를 작성하고 수정하고 테스트를 실행합니다.

주요 특징: Agent Computer Interface (ACI) — 모델이 코드와 상호작용하기 쉬운 인터페이스 설계. file_viewer, edit, search 등의 도구.

# Quick start
git clone https://github.com/princeton-nlp/SWE-agent
cd SWE-agent
pip install -e .

# Run on a GitHub issue
sweagent run \
  --config configs/default.yaml \
  --agent.model.name "claude-sonnet-4-6" \
  --problem_statement.github_url "https://github.com/..."

11 — Compliance & Licensing

합법적으로,
그리고 투명하게.

오픈소스 마켓의 본질은 신뢰. 자동 검출 + 옵트인 + 메타데이터 한정 호스팅의 3단 안전망으로 저작권 신고 위험을 사실상 제로화한다. 모든 사용자가 카드 한 장만 봐도 라이선스를 이해할 수 있게 설계됐다.

A. License Traffic Light · 자동 인덱싱 정책

● GREEN · AUTO-INDEX
✓
Permissive 라이선스
MIT · Apache 2.0 · BSD 2/3-Clause · ISC · Unlicense · CC0 · MPL 2.0

메타·README 발췌·태그 모두 자유롭게 표시. 가장 안전한 출발선이며 자동 인덱싱의 기본값.
● YELLOW · WARN BADGE
!
Copyleft 라이선스
GPL-2.0/3.0 · AGPL-3.0 · LGPL · CC BY 4.0

인덱스는 OK. 단 "Copyleft 의무" 또는 "Network Copyleft" 노란 경고 라벨을 카드에 함께 표시해 사용자가 도입 전 의무를 인지하도록.
● RED · EXCLUDED
×
제외 대상
라이선스 없음 · CC BY-NC/ND · BUSL · Elastic · Commons Clause · Custom · "Do not redistribute" 명시

자동 등록 거부. 사람이 검토 후 메인테이너 동의 받은 경우만 예외 승격.

B. 5 Safe Operating Principles · 안전 운영 5원칙

01
Metadata-Only Hosting
코드 본체는 절대 우리 서버에 저장하지 않음. harnesshub install은 실질적으로 GitHub clone. 우리는 디렉토리, 코드 저장소가 아님.
02
License Transparency
모든 카드에 SPDX 라이선스 뱃지를 항상 표시. YELLOW 등급은 추가 경고 라벨. 사용자가 클릭 전 라이선스를 본다.
03
Opt-In as Default
초기엔 수동 큐레이션 + 메인테이너 컨택. 점진적으로 GitHub Topic / harness.yaml 기반 자동화로 확장. 무단 등록 제로.
04
Maintainer-First Takedown
메인테이너 본인 인증(GitHub OAuth)이면 즉시 제거. 제3자 신고는 24~48h 내 검토 + 메인테이너 통지. DMCA 절차 완비.
05
Quote, Don't Copy
README 발췌 ≤200자 + 출처 링크. 이미지는 GitHub raw URL hot-link, 우리 CDN 저장 안 함. Fair use 라인 안에서.
★
Net Result
5원칙을 모두 지키면 npm·PyPI·crates.io와 동일한 법적 포지션. 검증된 모델로 운영 위험 최소화.

C. Detection Pipeline · 자동 검출 + 옵트인

5단계 분류 파이프라인

# Step 1: GitHub License API
GET /repos/{owner}/{repo}/license
→ SPDX ID 반환 시도

# Step 2: SPDX 매핑
GREEN list match  → GREEN
YELLOW list match → YELLOW
no SPDX           → continue

# Step 3: 파일시스템 검사
- LICENSE / LICENSE.md / COPYING 탐색
- go-license-detector 실행
- 모호 → RED

# Step 4: 텍스트 휴리스틱
README 키워드 스캔:
  "do not", "all rights reserved",
  "no commercial", "proprietary"
- 양성 → RED

# Step 5: 최종 분류
GREEN  → 자동 인덱스
YELLOW → 인덱스 + 경고 뱃지
RED    → 제외 + 운영진 큐

Opt-In Signals (Gold Standard)

라이선스 검사보다 더 확실한 건 메인테이너의 명시적 동의. 두 가지 방법으로 받는다.

METHOD A · GITHUB TOPIC

메인테이너가 자기 리포에 토픽 harnesshub 추가. 가장 가벼운 합의 표현.

METHOD B · harness.yaml
# 리포 루트에 추가 — 강력한 동의 신호
name: my-agent
license: MIT
maintainer: @username
opted_in: true

→ harness.yaml 추가 시 모델 호환성·설치 명령·벤치마크 점수 등 풍부한 메타 표시 가능.

통지 후 자동 인덱스

  • 메인테이너에게 알림 메일 발송
  • "등록됐습니다 · 빼려면 클릭" 1-click 옵트아웃

15 — Risks & Next Steps

예상 리스크와 대응.

시장에 진입하기 전 미리 정리한 4가지 위험과 그에 대한 답변. 검증되지 않은 가정도 명시한다.

⚠
"GitHub이 직접 만들면?"
GitHub Marketplace는 액션 중심. 하네스 도메인 특화 메타·검증·벤치마크는 따로 필요. 빠르게 디폴트 채널이 되는 것이 방어선.
⚠
스팸·저품질 등록
11번 섹션의 신호등 + 옵트인 + Verified 뱃지가 1차 방어선. 추가로 별 100+ / 최근 6개월 활성 / 실제 코드 존재 등 객관 신호로 디폴트 필터링.
⚠
하네스 정의 자체가 모호
"하네스 vs 라이브러리 vs 프레임워크" 경계 불분명. 초기엔 넓게 받고, 카테고리 구조로 정렬. 유저 검색 패턴 보며 정의 좁힘.
⚠
개발자 트래픽 부족
SEO + 콘텐츠 마케팅 (벤치마크 리뷰, 큐레이션) + Hacker News 런치 + Awesome 리스트 협업. 첫 한 달 1,000 명 무료 알파.
