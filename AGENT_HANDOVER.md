# 프로젝트 협업 워크플로우 및 환경 변경 안내

안녕하세요! 
이 문서는 여러 에이전트(AI 및 사용자)들이 코드가 엉키지 않고 협업할 수 있도록 새롭게 표준화된 프로젝트 구조와 규칙을 설명합니다. 프로젝트를 처음 파악하거나 이어받는 에이전트라면 반드시 이 가이드를 숙지하고 따라주세요.

## 1. 작업 디렉토리 단일화 (`agent-*` 폴더 제거)
기존에 에이전트 별로 분리해서 생성했던 `agent-A`, `agent-b-client` 등의 개별 디렉토리는 모두 삭제되었습니다. (이중 `.git` 레포지토리 문제 및 동기화 오류 방지)
**이제부터 모든 작업은 최상위 단일 루트 디렉토리(`./harness`) 하나에서만 진행합니다.**

## 2. 브랜치(Branch) 기반의 병렬 작업
작업 공간을 폴더로 쪼개는 대신, 각 에이전트는 Git 브랜치(Branch)를 활용하여 작업을 분리합니다.
* **작업 시작 시:** 항상 원격의 `main` 브랜치 최신본을 Pull 받은 뒤, `feature/agent-a-task` 와 같이 본인의 작업을 명확히 나타내는 브랜치를 새로 생성(Checkout)합니다.
* **작업 완료 시:** 해당 브랜치를 원격(Remote)에 Push하고, 메인(`main`)으로 병합(Merge)하여 전체 코드 베이스에 기여합니다.

## 3. 최신 Team-Architecture Harness 프레임워크 전면 도입
기존의 가상환경(`.venv`) 및 낡은 `.cursor/` 기반 하네스는 모두 폐기되었습니다. 이제부터는 [revfactory/harness](https://github.com/revfactory/harness) 플러그인이 프로젝트 내 `.claude/skills/harness`로 전역 탑재되어 있습니다. 

* **실행 방법**: 프로젝트에 새로운 요구사항이나 복잡한 구현을 시작할 때, 프롬프트로 언제든 **"하네스 구성해줘"** 또는 **"이 프로젝트에 맞는 에이전트 팀 구축해줘"** 라고 입력하세요.
* **동작 원리**: 하네스는 도메인을 자동으로 분석한 뒤, 파이프라인/계층적 위임 등 가장 적합한 아키텍처 패턴을 골라 최적화된 하위 에이전트 전문 팀(`.claude/agents/`)과 그들이 사용할 맞춤형 스킬(`.claude/skills/`)들을 작업 공간에 자동 생성해 줍니다.
* **규칙**: 모든 대규모 개발 혹은 복잡한 아키텍처 구현은 작업자의 자의적 판단으로 코딩하는 대신, 반드시 하네스를 먼저 구동하여 전문 에이전트 팀을 스폰(Spawn)한 뒤 위임하는 방식으로 안전하게 진행해야 합니다.

## 4. (예외 상황) 완전한 로컬 병렬 서버 구동이 필요한 경우
만약 같은 시스템 내에서 두 에이전트가 동시에 각기 다른 코드를 실행(`npm run dev`)하며 테스트해야 해서 **물리적으로 분리된 폴더가 필수적인 특수 상황**이라면?
* 수동으로 폴더를 복사(Clone)해서 만들지 **마세요**.
* Harness 프레임워크가 공식 지원하는 **Git Worktrees (`.cursor/worktrees.json`)** 기능을 사용하여 안전하게 병렬 작업 환경을 스폰(Spawn)해서 사용해 주세요.
